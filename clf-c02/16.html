雲端遷移：策略、工具與 CAF

Well-Architected Framework

On this page

遷移上雲 ​

雲端採用框架、七大遷移策略、資料庫與資料搬移工具。這章講「怎麼把既有的系統與資料從地端搬到 AWS」——重點是規劃框架（CAF）、七種搬法（7R），以及線上 / 離線搬資料的工具怎麼選。

為什麼遷移需要框架與策略 ​

把一整間公司的系統搬上雲，牽涉到組織、技能、安全、成本、相依關係——不是「複製貼上」這麼簡單。所以 AWS 提供兩個工具幫你想清楚：CAF（規劃的框架） 與 7R（每個應用怎麼搬的策略）。

AWS 雲端採用框架（Cloud Adoption Framework, CAF） ​

CAF 幫組織有系統地規劃與執行上雲，從六個面向檢視準備度。六個面向分成兩組：

商業能力（Business Capabilities）— 先想清楚「為什麼、誰來做」：

面向關注什麼
Business（商業）確保雲端投資對齊商業目標
People（人員）角色、技能、組織結構、教育訓練、變革管理
Governance（治理）管理與衡量雲端投資、專案 / 投資組合管理

技術能力（Technical Capabilities）— 再處理「怎麼建、怎麼守、怎麼跑」：

面向關注什麼
Platform（平台）建立 / 現代化雲端平台與架構
Security（安全）達成安全與合規目標
Operations（營運）運行、監控、維運雲端工作負載

前三個（商業 / 人員 / 治理）偏「人與管理」，後三個（平台 / 安全 / 營運）偏「技術與執行」。

七大遷移策略（The 7 Rs） ​

每一個既有應用程式，可以選七種搬法之一：

策略做什麼例子
Rehost原封不動搬上雲，不改架構（最快）把地端 VM 直接複製成 EC2
Replatform搬上雲時做小幅優化，不動核心架構把自管 MySQL 換成託管 RDS
Refactor / Re-architect重新設計成雲端原生（最費工、效益最大）把單體應用改成微服務 + 無伺服器
Repurchase換成另一個產品，常是 SaaS自架 CRM → 改用 Salesforce
Retain暫時保留在地端（還沒準備好 / 沒必要搬）即將汰除、或有相依限制的系統
Retire直接淘汰、移除不再需要的東西關掉沒人用的舊系統，省成本
Relocate把基礎設施整批搬到雲、不買新硬體、不改程式VMware Cloud on AWS
  • 「原封不動、最快搬上雲」→ Rehost
  • 「搬上雲順便換成託管服務（如 RDS）但不大改」→ Replatform
  • 「重新設計成雲端原生、投入最多」→ Refactor
  • 「換成 SaaS / 別的產品」→ Repurchase
  • 「這個系統先不搬」→ Retain
  • 「這個系統不要了」→ Retire

遷移服務與工具 ​

服務做什麼
AWS Migration Hub集中追蹤遷移進度的儀表板（跨各種遷移工具一覽）
AWS Application Migration Service（MGN）自動化 lift-and-shift（Rehost），把地端伺服器複製到 AWS
AWS Application Discovery Service盤點地端伺服器與相依關係，協助規劃遷移

資料庫遷移 ​

服務做什麼
AWS DMS（Database Migration Service）把資料庫搬到 AWS，搬遷期間來源資料庫仍可運作（降低停機）
AWS SCT（Schema Conversion Tool）來源與目標是不同引擎時，先轉換 schema
  • 同質遷移（Homogeneous）：來源與目標同引擎（MySQL → MySQL）→ 直接用 DMS
  • 異質遷移（Heterogeneous）：來源與目標不同引擎（Oracle → Aurora）→ 先用 SCT 轉換 schema，再用 DMS 搬資料

搬移資料：線上 vs 離線 ​

先問關鍵問題：網路頻寬夠不夠？

資料量 vs 頻寬
  頻寬充足、可接受傳輸時間  → 線上（走網路）
  頻寬不足 / 資料量巨大 / 偏遠無網路 → 離線（實體裝置寄送）

如果用現有網路傳要超過一週，就該考慮離線。例如 100TB 走 1Gbps 線路約要 12 天——這種就改用 Snow 裝置寄送更快。

線上傳輸 ​

服務做什麼
AWS DataSync自動化透過網路搬移大量資料（地端 ↔ AWS、或 AWS 儲存服務之間）
AWS Transfer Family用 SFTP / FTPS / FTP 把檔案傳進 / 傳出 S3、EFS
AWS Storage Gateway混合雲：地端 ↔ 雲端儲存的橋

離線傳輸：AWS Snow 家族 ​

裝置容量級距招牌特徵
Snowcone最小（8TB HDD / 14TB SSD）最輕最可攜，空間受限 / 嚴苛環境
Snowball EdgeTB ~ PB主力；分 Storage Optimized / Compute Optimized；可在裝置上跑 EC2 / Lambda 做邊緣運算
SnowmobileExabyte 級（最高 100PB）一台貨櫃車，搬整個資料中心
  • 「小、可攜、嚴苛環境」→ Snowcone
  • 「幾十 TB ~ PB、頻寬不足」→ Snowball Edge
  • 「EB 級 / 搬整座資料中心」→ Snowmobile

† Snowmobile（貨櫃車）已於 2024 年退役。Snowcone 已於 2024/11 停售。
