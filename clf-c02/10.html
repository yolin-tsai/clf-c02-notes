關聯式 vs 非關聯式（NoSQL） ​

關聯式資料庫（Relational / SQL）：資料放在資料表裡，每張表有固定的欄位（Schema），表格之間可以建立關聯（如「訂單表」用顧客編號連到「顧客表」），用 SQL 查詢。

像一本嚴謹的帳本——每一列都要照規矩填、欄位不能亂來，不同表格用編號互相對應。這種「嚴格 + 有關聯」的特性，讓它非常適合結構固定、彼此有關係、且不能出錯的資料：銀行交易、訂單系統、庫存。

非關聯式資料庫（NoSQL）：不要求固定 Schema，每一筆資料的結構可以不一樣；多半用「鍵 → 值」或「文件」方式存取，通常不做複雜的跨表關聯。

像一本可以隨意貼便利貼的筆記本——每張想寫什麼欄位都行，不必統一格式。犧牲嚴謹與關聯能力，換來極大規模的水平擴展與彈性。適合海量、結構多變、以簡單查找為主的場景：IoT 裝置資料、購物車、使用者 Session。

關聯式（SQL）非關聯式（NoSQL）
結構嚴格 Schema，資料表（列 + 欄）彈性，每筆可不同
關聯表格之間有關聯通常無複雜關聯
查詢SQL各家自己的 API
擴展寫入偏垂直；讀取可用 Read Replica 水平分擔偏水平（海量分散）
適合結構化、有關聯、交易（銀行、訂單）海量、彈性、簡單查找（遊戲、IoT）
AWS 服務RDS、AuroraDynamoDB

📌 資料結構固定、彼此有關聯、需要交易正確性 → 關聯式（RDS / Aurora）；海量規模、結構彈性、簡單快速查找 → NoSQL（DynamoDB）。

關聯式資料庫：Amazon RDS 與 Aurora ​

Amazon RDS（Relational Database Service） ​

RDS 是託管的關聯式資料庫。「託管」的重點在於：如果自己在 EC2 上裝資料庫，你得負責安裝、打補丁、備份、設定故障切換——很繁瑣。RDS 把這些雜事接手了，你專注用資料庫本身就好。

RDS 支援多種引擎：MySQL、PostgreSQL、MariaDB、Oracle、SQL Server，以及 AWS 自家的 Aurora。

兩個最常被搞混的機制：

機制目的怎麼運作
Multi-AZ（多可用區）高可用 / 故障自動切換在另一個 AZ 放一個 Standby（待命副本）；主資料庫故障時自動切換過去
Read Replica（唯讀副本）分擔讀取、提升讀效能建立唯讀副本承接大量讀取流量

Multi-AZ 的 Standby 平常不對外服務，它唯一的任務就是「主機掛了我接手」——是為了高可用，不是用來分散讀取。

  • 想提升讀取效能 / 分散讀取 → Read Replica
  • 資料庫故障要自動切換不中斷 → Multi-AZ

Amazon Aurora ​

Aurora 是 AWS 自己打造的關聯式資料庫，相容 MySQL 與 PostgreSQL，但效能與可用性更高：效能可達 MySQL 數倍，資料自動跨 3 個 AZ 存 6 份副本，儲存空間自動擴展。

可以把 Aurora 想成「RDS 家族裡的高階款」：要關聯式資料庫、又要更高效能與更強可用性，且能接受 MySQL / PostgreSQL 相容引擎時，選 Aurora。

NoSQL 資料庫：Amazon DynamoDB ​

DynamoDB 是完全託管的 NoSQL 資料庫（鍵值 / 文件型），特色是：

  • 無伺服器（Serverless）：沒有伺服器要管、不用做容量規劃
  • 任何規模都維持毫秒的回應：流量從一千暴增到一千萬，效能依然穩定
  • 自動擴展、自動跨多 AZ 高可用

💡 為什麼它能這麼快又這麼能擴展？因為它放棄了「跨表關聯 + 嚴格 Schema」的包袱，專注把「用鍵快速存取單筆資料」做到極致。代價是不適合需要複雜關聯查詢的場景。

適合：遊戲狀態、IoT、行動 App 後端、購物車、使用者 Session——「海量、要快、結構單純」的典型。

記憶體內快取（In-Memory Caching） ​

為什麼需要快取？ 假設網站首頁要從資料庫撈「熱門商品」，每秒上萬人都來看同一份資料——每次都去查資料庫又慢又貴。解法是把這份熱資料放進記憶體，下次直接拿（微秒級），大幅減輕資料庫負擔。

應用程式
    ↓ 先問快取（記憶體，微秒級）
ElastiCache ─── 有 → 直接回（超快，不打擾資料庫）
    ↓ 沒有（Cache Miss）
資料庫（RDS / DynamoDB）── 取出後存進快取，下次就有了
服務是什麼重點
Amazon ElastiCache託管的記憶體快取（Redis / Memcached）把熱資料放記憶體、加速讀取、減輕資料庫負擔
Amazon DynamoDB Accelerator（DAX）專給 DynamoDB 的記憶體快取讓 DynamoDB 的讀取從毫秒再降到微秒

其他資料庫服務 ​

不同用途的資料，有對應的專用資料庫：

服務類型適合
Amazon Redshift資料倉儲對海量歷史資料跑複雜分析、產報表（商業智慧 / BI）
Amazon Neptune圖形資料庫處理關係網絡：社群關係、推薦、詐騙偵測
Amazon DocumentDB文件資料庫相容 MongoDB 的文件型應用
AWS DMS（Database Migration Service）資料庫遷移把資料庫搬到 AWS，過程盡量不中斷

一般資料庫（RDS / DynamoDB）處理「日常交易」——一筆筆訂單、即時查詢，這類叫 OLTP（Online Transaction Processing，線上交易處理）；Redshift 是「資料倉儲」，把大量歷史資料集中起來跑複雜分析查詢，這類叫 OLAP（Online Analytical Processing，線上分析處理）。看到「資料倉儲、商業智慧、分析海量資料、產報表、OLAP」→ Redshift。

考試重點整理 ​

  • 第一刀分類：關聯式（嚴格 Schema、有關聯、SQL、交易）vs NoSQL（彈性、海量、鍵值查找）
  • RDS = 託管關聯式資料庫，支援 MySQL / PostgreSQL / MariaDB / Oracle / SQL Server / Aurora；接手安裝、補丁、備份、故障切換
  • 🚨 Multi-AZ vs Read Replica：Multi-AZ = 高可用 / 故障自動切換（Standby 平常不對外）；Read Replica = 分擔讀取 / 提升讀效能。別搞混
  • Aurora = AWS 自家高階關聯式（相容 MySQL / PostgreSQL），效能高、跨 3 AZ 存 6 份
  • 🚨 DynamoDB = 無伺服器 NoSQL，任何規模個位數毫秒、自動擴展、跨多 AZ；要 SQL / 關聯找 RDS，要海量鍵值找 DynamoDB
  • 快取：ElastiCache（Redis / Memcached，加速讀取、減輕資料庫）；DAX（專給 DynamoDB 的記憶體快取，降到微秒）
  • 🚨 Redshift = 資料倉儲（OLAP）；RDS / DynamoDB 處理日常交易（OLTP）；看到「BI、分析海量歷史資料、OLAP」→ Redshift
  • Neptune（圖形 / 關係網絡）、DocumentDB（文件 / MongoDB 相容）、DMS（資料庫遷移上雲）
情境需求答案
結構固定、有關聯、交易正確（訂單、銀行）RDS
要關聯式、又要更高效能與可用性Aurora
海量、彈性 Schema、極快鍵值查找、無伺服器DynamoDB
資料庫故障要自動切換、不中斷RDS Multi-AZ
想分擔讀取流量、提升讀效能RDS Read Replica
加速重複讀取、減輕資料庫壓力ElastiCache
加速 DynamoDB 的讀取（到微秒）DAX
對海量資料跑分析、做 BI 報表Redshift（資料倉儲）
處理社群關係 / 推薦 / 詐騙這類關係網絡Neptune（圖形）
把現有資料庫搬上 AWSDMS
