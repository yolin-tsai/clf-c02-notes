雲端安全 ​

IAM、加密、威脅偵測與合規。這章服務最多，重點在分清楚每個服務解決什麼問題。

責任共擔模型 ​

AWS 管「雲本身的安全」（Security OF the Cloud），你管「雲裡面的安全」（Security IN the Cloud）。

你的資料、你的 IAM 權限設定，永遠是你的責任，不管用哪種服務。這章講的所有工具，都是幫你做好「IN the Cloud」這一半。

IAM（Identity and Access Management） ​

IAM 管「誰（身分）能對哪些資源做什麼（權限）」。可以想成公司的門禁系統：發識別證、分部門、訂每個人能進哪些門。

IAM 是全球服務（Global），不綁特定 Region——你建的使用者在所有 Region 都通用。

最小權限原則（Principle of Least Privilege） ​

只給「完成工作所需的最小權限」，不多給一點。

這是整個 IAM 設計的核心思想：

  • 起點是零：新建 User / Role 預設什麼都做不了，需要什麼才加什麼，而不是先給到飽再慢慢收
  • 縮小爆炸半徑（Blast Radius）：帳號萬一被盜，攻擊者只能做那個帳號權限範圍內的事——授權限得愈小，損失愈小
  • 避免萬用字元：IAM Policy 寫 "Action": "*" 代表「什麼操作都能做」，看起來方便，實際上是最容易出問題的設定；應精確指定實際需要的動作

IAM 核心元件 ​

元件是什麼類比
Root User（根使用者）用 email 註冊帳號時建立的最高權限帳號創辦人的萬能鑰匙
IAM User（使用者）給「一個人或應用」的長期身分員工識別證
IAM Group（群組）一群使用者的集合，套同一組權限部門
IAM Policy（政策）用 JSON 寫的權限清單，定義允許/拒絕什麼權限明細表
IAM Role（角色）臨時憑證，不綁特定人，誰需要誰「擔任」臨時通行證

Root User：建立後就鎖起來 ​

Root User 權力過大（可關帳號、改帳單），最佳實踐是日常絕對不要用它。正確做法：

  1. 立刻幫 Root User 開 MFA（多因素驗證）†
  2. 建立 IAM 使用者來做日常工作
  3. Root User 只在極少數必須的操作（如改帳戶設定）才動用

† 自 re:Inforce 2025 起，AWS 對所有帳號的 root user 強制啟用 MFA——現為平台要求，不只是最佳實踐。

IAM User vs IAM Role ​

兩者都能持有權限，差別在「有沒有長期憑證」：

IAM UserIAM Role
對象固定的一個人 / 應用不綁人，誰需要誰臨時擔任
憑證長期帳密 / 存取金鑰臨時、會自動過期
典型場景工程師登入 ConsoleEC2 要存取 S3、跨帳號存取、第三方聯合登入

「EC2 上的應用程式要存取 S3，該怎麼做？」→ 指派一個 IAM Role 給 EC2，不要把存取金鑰寫死在程式或主機裡。（用角色拿臨時憑證，不用長期金鑰。）

MFA（Multi-Factor Authentication，多因素驗證） ​

帳密只是「你知道的東西」，一旦外洩攻擊者立刻能登入。MFA 在密碼之後加一個「你擁有的東西」，兩個因素都對才能進去：

登入流程：輸入密碼 → 再提供動態驗證碼（6 位數、30 秒換一次）→ 才能成功登入

AWS 支援三種 MFA 裝置：

類型說明代表例子
Virtual MFA（虛擬 MFA）手機 App 每 30 秒產生一組動態碼（TOTP）Google Authenticator、Authy
Hardware TOTP token實體小裝置，每 30 秒顯示一組數字第三方廠商實體 token
FIDO Security Key插 USB 或 NFC 輕碰的硬體金鑰YubiKey

Root User 必須開 MFA：Root User 是整個 AWS 帳號的最高權限，設定完就要鎖起來不日常使用，並立刻開啟 MFA。有管理員權限的 IAM User 也強烈建議開啟。

IAM Identity Center（原 AWS SSO） ​

當你有很多 AWS 帳號 / 很多使用者時，用 IAM Identity Center 做單一登入（SSO），一處集中管理，還能接公司既有的目錄（如 Active Directory）。

AWS Organizations 與 SCP ​

企業用雲一段時間後，往往會有多個 AWS 帳號——開發、測試、生產各一個，或不同部門、子公司各自的帳號。AWS Organizations 就是把這些帳號集中到一個組織下統一管理的服務。

為什麼要分帳號？ 帳號是 AWS 最強的安全隔離邊界——同帳號內的資源彼此「看得到」，不同帳號預設完全隔離。大組織讓不同環境（dev / prod）或不同業務單位各自一個帳號，Organizations 再把這些帳號「管起來」。

Organizations 可以把帳號分組成 OU（Organizational Unit，組織單位），像一棵樹——你可以在不同 OU 套不同的管控政策：

Root（根）
├── OU：生產環境
│   ├── 帳號 A（Web 服務）
│   └── 帳號 B（後端服務）
├── OU：開發環境
│   └── 帳號 C（測試）
└── OU：安全與日誌
    └── 帳號 D（集中記錄）

合併帳單（Consolidated Billing）：Organizations 也能把所有帳號的費用合進一張帳單，還能讓多帳號用量合計享量大折扣——詳見帳單那章。

SCP（Service Control Policy，服務控制政策） 設的是「權限上限 / 邊界」，不是直接授權。例如用 SCP 規定「這個 OU 底下所有帳號都不准用 us-east-1 以外的 Region」。即使某帳號的 IAM Policy 給了更多權限，也會被 SCP 的上限擋住。

最終權限 = IAM 給的 ∩ SCP 允許的

† RCP（Resource Control Policies，資源控制政策）：2024 年新增的姊妹機制。SCP 管「身分能做什麼動作的上限」，RCP 管「資源能被誰存取的上限」——兩者合力從不同維度設天花板。CLF 知道有這個機制即可；細節偏 SAA。

IAM vs Security Group／NACL ​

兩個都叫「安全」，守的卻是不同的東西

讀到這裡，AWS 有兩組「安全規則」：這章的 IAM，和前面學過的 Security Group／NACL。它們常被混在一起——都叫「安全」、都用「允許／拒絕」的語言。但它們其實守在完全不同的兩層。

它們各自回答的，是兩個毫不相干的問題：

  • IAM 回答：「這個身分，能不能對 AWS 下某個操作指令？」——例如能不能呼叫 API 去建立、讀取、刪除某個 AWS 資源。它在乎的是「你是誰、被授權做哪些動作」，屬於控制台／API 這一層。
  • Security Group／NACL 回答：「這個網路封包，能不能抵達或離開某台機器／子網？」——它根本看不到你的 AWS 身分是誰，只看封包的來源 IP、目的 port、協定，屬於網路流量這一層。

封包能不能到達 EC2 的 port 22 → Security Group／NACL 說了算（網路層）

你能不能在 AWS 裡把這台 EC2 建立、停止、終止 → IAM 說了算（API 層）

關鍵在於兩者互不替代：一個 IAM 權限開到最大的管理員，如果 Security Group 沒放行 22，一樣 SSH 不進去；反過來，Security Group 全開，也不代表某個 IAM 使用者就有權呼叫 API 把這台機器關掉。它們各守一層，缺一不可。

比較IAM（＋ SCP）Security Group／NACL
守的層面AWS 控制台／API：能對 AWS 資源做什麼操作網路流量：封包能不能抵達機器／子網
判斷依據你是誰、被授權哪些動作封包的來源 IP／port／協定
看得到你的 AWS 身分嗎是（身分就是它的核心）否（只看封包）
典型問題「能不能呼叫 API 讀這個 S3 bucket？」「HTTP 封包能不能連到這台 Web server？」

SCP 雖然也能「拒絕」，但它拒絕的是帳號能執行哪些 API 動作，跟 NACL 拒絕網路封包是不同層。SCP 其實跟 IAM 同一邊（都在管「身分／動作」），只是它管的是整個帳號的權限上限。

資料加密與金鑰管理 ​

服務功能使用時機
KMS（Key Management Service）AWS 託管的加密金鑰服務大多數加密需求的標準答案
CloudHSM專用硬體安全模組（HSM），金鑰完全由你掌控法規硬性要求專屬硬體時使用
ACM（Certificate Manager）配發 / 管理 SSL/TLS 憑證要讓網站跑 HTTPS、幫 ALB / CloudFront 上憑證
Secrets Manager安全存放密碼 / API 金鑰，可自動輪替程式要用 DB 密碼、又想定期自動換

威脅偵測與調查 ​

這五個服務最常一起出現在選項裡：

服務一句話抓什麼 / 範圍
GuardDuty威脅偵測分析 VPC Flow Logs、DNS、CloudTrail 日誌，找惡意/異常行為
Inspector漏洞掃描掃 EC2 / 容器映像 / Lambda 的軟體漏洞與不當設定
Macie敏感資料發現用 ML 找 S3 裡的 PII / 敏感資料（限 S3）
Security Hub集中儀表板彙整 GuardDuty / Inspector / Macie 等的發現到單一視圖
Detective根因調查把發現的安全問題深入分析、找根本原因
  • 抓「惡意行為 / 未授權存取」（看日誌找壞人）→ GuardDuty
  • 抓「漏洞 / 該打的修補程式 / CVE」→ Inspector
  • 找「S3 裡的個資 / PII / 敏感資料」→ Macie
  • 要「跨帳號集中看所有安全發現」→ Security Hub
  • 要「調查事件根本原因」→ Detective

網路與應用層防護 ​

服務防什麼重點
ShieldDDoS 攻擊Standard 自動啟用、免費、人人有；Advanced 付費，含 24/7 支援與成本保護
WAF（Web Application Firewall）L7 網頁攻擊依規則過濾 HTTP/HTTPS，擋 SQL Injection、XSS
Network FirewallVPC 層級網路流量對整個 VPC 做網路流量過濾
Firewall Manager跨帳號集中管理統一管理多帳號的 WAF / Shield 規則

其中 Shield 分兩個等級，差別非常大：

Shield StandardShield Advanced
費用免費，自動啟用付費（月費 + 流量費）
申請方式不需要，人人都有需主動訂閱
防護對象所有 AWS 資源（自動）EC2、ELB、CloudFront、Route 53、Global Accelerator
攻擊類型常見 L3/L4 DDoS（流量淹沒、SYN flood…）同上，加上更複雜的攻擊
DDoS 應變團隊（DRT）無24/7 專人支援
費用保護無DDoS 導致資源暴增的帳單可申請退款
進階監控與告警無有（近即時可見度）

什麼時候需要 Shield Advanced？ 你的服務是高知名度、高流量的公共網站或應用——例如電商大促、遊戲發布、金融服務——且需要 AWS 的 DDoS 應變團隊（DRT）在攻擊時直接介入協助、事後還能追討被灌爆的帳單，這才值得訂 Advanced。一般情境 Standard 就夠了。

📌 WAF vs Shield：擋「應用層攻擊（SQLi / XSS）」→ WAF；擋「流量灌爆的 DDoS」→ Shield。

合規（Compliance） ​

服務 / 概念是什麼
AWS Artifact下載 AWS 的合規報告（SOC、PCI DSS、ISO 等），交給稽核單位看
AWS Audit Manager自動蒐集證據、協助你跑稽核流程

Artifact 只是「拿現成的合規報告」的地方，它不會幫你做稽核、也不監控你的設定。要「追蹤設定變更是否合規」是 AWS Config （下一章介紹）。

考試重點整理 ​

  • 責任共擔：AWS 管雲本身（OF），你管雲裡面（IN）；資料與 IAM 設定永遠是你的責任
  • IAM 是全球服務；五元件：Root User（鎖起來別用、開 MFA）、User（人）、Group（部門）、Policy（JSON 權限）、Role（臨時憑證）
  • User vs Role：要長期身分用 User；EC2 / 服務 / 跨帳號要存取資源用 Role（臨時憑證，不寫死金鑰）
  • 最小權限原則 + MFA 是 IAM 最佳實踐的兩大關鍵字
  • SCP（Organizations）設「權限上限」，不是授權；最終權限 = IAM ∩ SCP
  • 加密：KMS（標準）、CloudHSM（法規要專屬硬體才用）、ACM（SSL/TLS 憑證）、Secrets Manager（密碼自動輪替）
  • 威脅偵測：GuardDuty（惡意行為）、Inspector（漏洞）、Macie（S3 個資）、Security Hub（集中彙整）、Detective（查根因）
  • 網路防護：Shield（DDoS，Standard 免費自動）、WAF（L7，擋 SQLi/XSS）、Network Firewall（VPC 流量）、Firewall Manager（跨帳號管理）
  • 合規：Artifact = 下載報告（不做稽核/監控）
