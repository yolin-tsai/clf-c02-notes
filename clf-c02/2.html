Amazon EC2 是什麼 ​

EC2（Elastic Compute Cloud）提供「安全、可調整大小的雲端運算容量」——就是在雲上租一台虛擬伺服器。你可以在上面裝任何作業系統與軟體，跑網站、跑應用程式、跑資料庫。

對比傳統做法就懂它的價值：自建機房要先採購硬體、等貨到、上架、裝設定，動輒數週；EC2 幾分鐘就能開好一台，不用了就關掉、只付用到的時間（呼應雲端「變動支出取代固定支出」「提升速度與敏捷」）。

怎麼運作（三步）：

1. 啟動（Launch）：選一個 AMI（系統範本）＋ 選 instance type（硬體規格）
2. 連線（Connect）：用 SSH（Linux）或 RDP（Windows）連進去
3. 使用（Use）：裝軟體、跑你的應用

💡 同一台實體主機上，AWS 用虛擬化切出多台 EC2 給不同客戶，彼此隔離互不干擾。這就是「多租戶」——你感覺像獨佔一台機器，實際上是共享底層硬體。

EC2 執行個體類型 ​

AWS 依「強化哪一種硬體」把機器分成五大類別，讓你按工作的瓶頸挑機器。不必背型號，記住每一類強化什麼、適合什麼即可：

類型優化方向適合場景
General Purpose（通用型）運算 / 記憶體 / 網路均衡網站、應用伺服器、中小型資料庫——不確定時的預設選擇
Compute Optimized（運算優化）高 CPU高效能運算（HPC）、批次處理、遊戲伺服器、ML 推論
Memory Optimized（記憶體優化）大記憶體大型記憶體內資料庫、即時處理大量資料
Storage Optimized（儲存優化）高速本地儲存、高 IOPS資料倉儲、分散式檔案系統、需要大量本地讀寫的工作
Accelerated Computing（加速運算）GPU / 專用晶片機器學習訓練、繪圖渲染、大量浮點運算
  • 看到「需要大量記憶體跑記憶體內資料庫」→ Memory Optimized；「機器學習訓練 / GPU」→ Accelerated Computing；「不確定 / 一般網站」→ General Purpose。
  • 每個「類別」底下還有很多家族（family）與世代（如 m7i、c7g…），不必背型號，認得字首對應哪一類就好。

開機前要懂的三個東西 ​

  • AMI（Amazon Machine Image，系統映像） — 預先打包好的「作業系統 + 預裝軟體」範本。選 AMI 等於決定「這台機器一開機長什麼樣」（例如 Amazon Linux、Windows Server，或已裝好 WordPress 的映像）。
  • Key Pair（金鑰對） — SSH 登入用的「鑰匙」。AWS 給你一個私鑰檔，連線時用它證明身分。務必保存好，弄丟無法重新下載。
  • Security Group（安全群組） — 這台 EC2 的防火牆（呼應先備知識）。預設把連入流量全擋掉，你要手動開放需要的 port（SSH 22、HTTP 80）。

執行個體狀態與生命週期 ​

開好的 EC2 不是只有「開」和「刪」，幾個動作的差別直接關係到會不會繼續被收費：

動作發生什麼還在收費嗎
Stop（停止）關機，保留根磁碟與資料，可再開機運算費停收，但 EBS 磁碟仍收費
Reboot（重開機）重新開機，資料與設定不變照常
Hibernate（休眠）把記憶體內容存到 EBS，下次開機回到原狀態EBS 仍收費
Terminate（終止）刪除執行個體，預設一併刪掉根 EBS運算停收，但殘留資源可能還在收費
  • Stop 不等於不收費——它像「熄火但車還停在收費停車場」，EBS 磁碟持續計費。真的不用了要 Terminate，而且 Terminate 也不一定清乾淨：額外的 EBS、快照、Elastic IP、負載平衡器都可能殘留繼續收費。練習完務必回頭檢查並刪除。
  • EC2 需要儲存：每台 EC2 都有一顆根磁碟（通常是 EBS），你的資料存在上面。

與 AWS 互動的方式 ​

在 AWS 上做的每一件事，本質都是一個 API 呼叫

差別只在「你用什麼方式發出這個呼叫」。

方式是什麼適合
管理主控台（Management Console）網頁圖形介面，點選操作初學、手動操作、視覺化探索
CLI（命令列介面）用指令操作，可寫成腳本自動化、重複性操作
SDK在你的程式碼裡呼叫 AWS（Python、Java…）應用程式直接整合 AWS
CloudFormation基礎設施即程式碼（IaC）：用範本宣告要哪些資源可重複、一致地部署整套環境
Elastic Beanstalk你給程式碼與設定，它幫你自動佈建 EC2 / ELB / Auto Scaling不想手動拼裝基礎設施

「點選的網頁介面」→ Console；「可寫腳本自動化」→ CLI；「用範本重複部署、基礎設施即程式碼」→ CloudFormation。

EC2 計費 ​

同一台機器，怎麼付錢有五種選擇，差別在「有沒有事先承諾」與「能不能被收回」：

計費方式怎麼算折扣適合
On-Demand（隨需）用多少付多少，無承諾無短期、不可預測、開發測試
Savings Plans（節省方案）承諾 1 或 3 年的固定用量（以 $/小時計）大用量穩定、想省錢又保留彈性
Reserved Instances（預留執行個體）承諾 1 或 3 年特定規格大長期穩定、可預測的負載
Spot Instances（競價執行個體）用 AWS 的閒置容量最高約 90%可中斷的工作：批次、無狀態、容錯
Dedicated Hosts（專屬主機）整台實體主機專屬給你，可看到主機 ID、sockets / cores—BYOL（自帶授權）或法規要求掌握實體硬體資訊

  • Spot 最便宜，但 AWS 需要容量時會把它收回（給你約 2 分鐘警告）。所以只放可以被中斷、之後能重跑的工作——絕對不要把「不能中斷的關鍵服務」放在 Spot 上。

  • Savings Plans vs Reserved：兩者都是「承諾換折扣」。Savings Plans 較有彈性（承諾的是金額 / 用量）；Reserved 綁較死（承諾特定規格）。

  • Dedicated Instances vs Dedicated Hosts：兩者的硬體都不和其他客戶共用。差別在於：Dedicated Instances 由 AWS 管理主機放置，你看不到底層實體機器；Dedicated Hosts 把整台實體主機交給你，你能看到 socket / core 數量、控制執行個體放哪台主機——這是 BYOL（Bring Your Own License，自帶授權） 的必要條件，因為 Windows Server、SQL Server 等授權是依實體 socket / core 計費。

  • 看到「BYOL / 把現有軟體授權帶到 AWS」→ Dedicated Hosts；只是要「實體隔離但不在意哪台主機」→ Dedicated Instances（較便宜）。

EC2 Auto Scaling ​

Auto Scaling 讓 EC2 依需求自動增減數量。這直接呼應雲端的「不必預估容量」：傳統機房得照尖峰常備一堆平常閒置的機器；Auto Scaling 則是流量大自動加機器、流量小自動移除，省錢又不當機。

水平擴展 vs 垂直擴展：Auto Scaling 做的是水平擴展（scale out，加減機器數量），不是把單一台機器升級成更強的（垂直擴展 scale up）。

垂直（升級單機）水平（加減機器）
規模上限有天花板近乎無上限
升級時通常要停機不停機
故障風險單點故障，一掛全倒個別機器掛了還有別台

兩種擴展策略：

策略說明
動態擴展（Dynamic）依當下指標（如 CPU 使用率）即時增減
預測擴展（Predictive）依歷史模式預測未來需求，提前準備

設定時會定義三個值：最小（minimum）／期望（desired）／最大（maximum） 數量——確保至少有幾台、平常維持幾台、最多長到幾台。

Elastic Load Balancing（ELB） ​

ELB 是託管的負載平衡器（Load Balancer），把進來的流量平均分配到多台 EC2，並且只把流量導向健康的機器。若有一台 EC2 當掉了、ELB 卻還把流量送過去，那些使用者就會收到錯誤。健康檢查（Health Check）讓 ELB 持續探測每台機器，自動跳過故障的那台，使用者完全無感——這正是高可用（High Availability）的關鍵。

為什麼 ELB 和 Auto Scaling 是天生一對： Auto Scaling 負責「該有幾台機器」，ELB 負責「流量怎麼平均分到這些機器」。機器增加時，ELB 自動把新機器納入分流；機器移除或故障時，ELB 自動把它從分流名單拿掉。

              使用者流量
                  ↓
    Elastic Load Balancing（只分流到健康的機器）
       ↓            ↓            ↓
     EC2          EC2          EC2       ← Auto Scaling 群組
 （流量大 → 自動加機器；流量小 → 自動移除）

重點整理 ​

  • EC2 = 雲上可調整大小的虛擬伺服器；幾分鐘開好、用多少付多少；底層用虛擬化做多租戶隔離
  • 執行個體類別：General Purpose（均衡）、Compute Optimized（高 CPU）、Memory Optimized（大記憶體）、Storage Optimized（高速本地儲存）、Accelerated Computing（GPU）
  • 開機三要素：AMI（系統映像）、Key Pair（SSH 金鑰）、Security Group（防火牆，需手動開 port）
  • 佈建方式：Console（網頁手動）、CLI（腳本）、SDK（程式碼）、CloudFormation（IaC、可重複部署）、Elastic Beanstalk（自動佈建）
  • 計費：On-Demand（無承諾最彈性）、Savings Plans / Reserved（承諾 1–3 年換折扣）、Spot（閒置容量最省約 90%、可被收回）、Dedicated Hosts（專屬實體機、為法規 / 授權）
  • 水平 vs 垂直：垂直有天花板 + 要停機 + 單點故障；水平近乎無上限 + 高可用——雲端偏好水平
情境需求答案
短期、不可預測、開發測試的負載On-Demand
長期穩定、可預測、想省錢Savings Plans / Reserved Instances
可中斷的批次工作、要最省錢Spot Instances
法規 / 軟體授權要求專屬實體主機Dedicated Hosts
流量起伏大，要自動增減機器EC2 Auto Scaling
把流量平均分配到多台健康主機Elastic Load Balancing
用範本可重複部署整套基礎設施CloudFormation（IaC）
需要大記憶體跑記憶體內資料庫Memory Optimized 執行個體
