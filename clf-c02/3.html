訊息與佇列：SQS 與 SNS ​

當應用程式拆分為多個獨立元件，元件之間需要一種可靠且互不干擾的通訊機制——這正是訊息服務解決的核心問題。

  • 耦合性（Coupling）

    • Day-05: 耦合性三大原則 ADP、SDP、SAP

  • 生產者－消費者模式（Producer-Consumer Pattern）

    • iThome：Day 28 介紹 Producer Consumer 模式
    • Medium：Java Concurrency — 生產者消費者模式，解耦合的極致

為什麼需要訊息服務 ​

現代應用架構的演進路徑，本質上是一場降低耦合的工程：

  • 緊耦合（Monolithic）：所有功能整合在單一應用中，元件之間直接同步呼叫。任一元件發生故障，可能引發連鎖反應，導致整體服務中斷。
  • 鬆耦合（Microservices）：元件之間透過非同步訊息通訊，相互獨立，不直接綁定。

將元件解耦後，架構獲得兩項關鍵特性：

  1. 故障隔離（Fault Isolation）：下游元件不可用時，訊息暫存於佇列，待其恢復後繼續處理，不影響上游的正常運行。
  2. 負載緩衝（Load Leveling）：上游流量驟增時，訊息先進入佇列排隊，下游依自身處理速率逐一消化，不會因瞬間流量峰值而過載。

SQS 與 SNS 是實現鬆耦合架構的兩個核心服務。

以電商「結帳」為例 ​

顧客送出訂單，後端同時觸發兩條路徑：

① 訂單寫入 SQS 佇列 ──→ 倉儲系統依自身速率逐一取出處理
   （一秒湧入萬筆訂單時，佇列作為緩衝，倉儲系統不因此過載）
   → 負載緩衝 + 故障隔離

② 發布「訂單成立」事件至 SNS 主題 ──→ 同時推送給多個下游訂閱者
   （寄送確認信、通知財務系統、通知物流）
   → 一對多 fan-out

SQS 處理「單一任務交付單一消費者、確保每則訊息都被處理」；SNS 處理「單一事件同步廣播至多個接收者」。

Amazon SQS（Simple Queue Service） ​

採用拉取（Pull）模型：訊息進入佇列排隊，消費者在就緒時主動發起 poll 請求取得訊息，處理完成後才從佇列刪除。這種設計確保每則訊息至少被處理一次，且消費者的處理速率不受上游產生速率影響。

核心機制：

機制說明
訊息保留期限訊息最長保留 14 天（預設 4 天），待消費者取用
可見性逾時（Visibility Timeout）訊息被取出後進入隱藏狀態，防止其他消費者重複領取；處理完成確認後才正式刪除
死信佇列（Dead-Letter Queue, DLQ）訊息達到最大重試次數後仍處理失敗，自動移入 DLQ 隔離，便於後續診斷與排查

佇列類型：

類型特性
Standard近乎無限吞吐量；盡力維持順序，允許偶發的亂序與極少數重複訊息
FIFO（先進先出）嚴格保證訊息順序與僅處理一次（Exactly-Once Processing），適用於金融交易等順序敏感場景

Amazon SNS（Simple Notification Service） ​

採用發布 / 訂閱（Pub/Sub）模型：發布者將訊息送至主題（Topic），SNS 立即以推送（Push）方式將訊息同時分發至所有訂閱者，實現一對多的 fan-out 通訊。

支援的訂閱者類型： Email、SMS 簡訊、HTTP/S 端點、AWS Lambda、SQS 佇列、行動裝置推播（APNs / FCM）。

                    ┌─→ Email（寄送訂單確認信）
發布者 →[ SNS 主題 ]─┼─→ Lambda（觸發後續自動化流程）
                    └─→ SQS 佇列（轉交下游系統非同步處理）
        （單一發布，所有訂閱者同步接收）

🚨 看到「單一事件需同時通知或觸發多個接收者」→ SNS。

SQS vs SNS 對照 ​

Amazon SQS（佇列）Amazon SNS（發布訂閱）
通訊模型佇列：訊息排隊等候消費Pub/Sub：發布即分發至所有訂閱者
訊息流向消費者主動拉取（Poll）SNS 主動推送（Push）至訂閱者
接收者數量通常一個消費者處理一則訊息一對多，所有訂閱者同時接收
典型用途任務佇列、突發流量緩衝、確保訊息不遺漏事件通知（Email/SMS）、一個事件觸發多個下游動作

Fan-out 模式（SNS + SQS） ​

以 SNS 將單一事件推送至多個 SQS 佇列，每個下游系統擁有獨立佇列，各自按自身速率處理、互不干擾。

                       ┌─→ [SQS] → 倉儲系統
訂單成立 →[ SNS 主題 ]──┼─→ [SQS] → 財務系統
                       └─→ [SQS] → 數據分析管線
   （單次發布；三個佇列各保有一份副本，各自獨立消費）

為何不讓 SNS 直接推送至各下游系統，而要中間插入 SQS？因為佇列同時提供了緩衝與重試能力——某個下游暫時不可用時，訊息持續保留在其佇列中，待服務恢復後繼續處理，對其他下游系統完全無影響。

重點整理 ​

  • 核心區別：SQS = 拉取、點對點、一則訊息由一個消費者處理（任務緩衝）；SNS = 推送、Pub/Sub、一對多同步分發（事件廣播）。
  • 兩者皆為鬆耦合架構工具，共同帶來故障隔離與負載緩衝兩項架構優勢。
  • SQS 關鍵細節：訊息最長保留 14 天；可見性逾時防止重複消費；DLQ 隔離持續失敗的訊息；Standard（高吞吐、盡力排序）vs FIFO（嚴格順序、Exactly-Once）。
  • SNS 訂閱者：Email / SMS / HTTP-S / Lambda / SQS / 行動推播。
情境需求答案
任務排隊、確保每則訊息都被處理SQS
突發流量湧入，需緩衝避免下游過載SQS
訊息處理順序不可錯亂SQS FIFO
單一訊息同時通知多個接收者SNS
訂單成立後同時通知倉儲、財務、EmailSNS（fan-out）
廣播事件且每個下游需獨立緩衝與重試SNS + 多個 SQS（fan-out 組合）
持續處理失敗的訊息需隔離排查死信佇列（DLQ）
