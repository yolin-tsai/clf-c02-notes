容器：ECS、EKS、Fargate 與 ECR ​

為什麼要用容器：先看它和虛擬機的差別 ​

要理解容器，最快的方式是拿它跟上一章學的 EC2（虛擬機） 比。

一台虛擬機（VM）是「一整台電腦的數位複本」——它有自己完整的作業系統，跑在實體主機的作業系統之上。所以每開一台 VM，都要扛一份完整 OS 的重量：開機慢、佔空間、資源要事先備好。

容器則只打包「你的程式 + 它需要的相依套件」，然後共用底層主機的作業系統核心。因為不必每次都搬一整套 OS，容器輕量、啟動快、好搬移。

用貨櫃來想「容器（container）」這個名字：貨櫃把貨物用一個標準化的箱子裝好，不管換到哪艘船、哪台卡車都能直接搬、不必重新打包。容器對軟體做的是同一件事——把程式連同環境裝進一個標準單位，在你的筆電、在測試機、在雲上跑起來都一模一樣，解決「明明我電腦上可以跑」的問題。

虛擬機（如 EC2）容器
打包什麼一整套作業系統 + 應用只有應用 + 相依套件
作業系統每台各有一份完整 OS共用主機的 OS 核心
啟動速度較慢（要開機）極快
資源佔用較重輕量
隔離強度強（完整隔離）相對輕（行程層級隔離）†

† Fargate 是例外：每個 task 跑在自己的 Firecracker micro-VM 裡，隔離強度接近 VM 級，而非行程層級。

容器和虛擬機不是二選一。最常見的是「容器跑在虛擬機裡」——一台 EC2 上面跑很多容器。

映像檔與容器 ​

  • 映像檔（Image） 是「範本」——一個唯讀的打包成品，裡面是程式加上它要的環境。可以想成「餅乾模具」。
  • 容器（Container） 是「用範本跑起來的實體」——由映像檔啟動的執行中程式。可以想成「用模具壓出來的餅乾」，一個模具可以壓出很多片。

流程是：先做出映像檔 → 存到倉庫 → 需要時從映像檔啟動一或多個容器。那個「倉庫」在 AWS 上就是稍後會講的 ECR。

編排（Orchestration） ​

一兩個容器你還能手動顧。但當你有幾十、幾百個容器時，一堆問題冒出來：哪個容器該跑在哪台機器？某個掛了要不要自動重啟？流量變大要不要多開幾個、變小要不要收掉？多台機器之間怎麼分配？

替你自動處理這些事的系統，就叫編排器（Orchestrator）。 它是容器世界的「調度中心」。在 AWS 上跑容器，核心就是兩個要分開決定的問題。

決定一：用哪個編排器 ​

編排器是什麼選它的理由
ECS（Elastic Container Service）AWS 原生的容器編排服務較簡單、與 AWS 服務整合最好、控制平面免費
EKS（Elastic Kubernetes Service）AWS 託管的 Kubernetes開源業界標準、可跨雲 / 地端移植、團隊已熟 K8s

ECS 是 AWS 自己設計的編排器。它的控制平面（決定容器怎麼調度的大腦）完全由 AWS 管，你不必碰，所以上手最快、和 IAM、ALB、CloudWatch 等 AWS 服務接得最順。代價是它是 AWS 專屬的，綁在 AWS 生態裡。

EKS 則是把 Kubernetes（業界最主流的開源容器編排標準，常簡寫 K8s）交給 AWS 託管。選它的最大理由是可攜性與標準化：同一套 Kubernetes 設定，可以在 AWS、其他雲、甚至自家機房用同樣的工具跑。代價是 Kubernetes 本身較複雜，要懂它的一套術語與運作。

ECS vs EKS：要 AWS 原生、簡單、整合好 → ECS；要開源 Kubernetes、跨雲可攜、團隊已用 K8s → EKS。

決定二：容器實際跑在哪 ​

計算選項誰管主機你要負責什麼
EC2 啟動類型你自己準備 EC2、修補更新、規劃容量與擴展
FargateAWS（無伺服器）只管容器本身，主機完全不用碰
  • EC2 啟動類型：容器跑在你自己管理的一群 EC2 上。你握有最多掌控權（可挑機型、可裝特殊東西、用量穩定時可能更省），但也要自己負責這些主機的修補、容量與擴展。
  • Fargate：無伺服器模式。你只要說「我要跑這個容器、給它多少 CPU / 記憶體」，AWS 就替你在背後準備好主機來跑，你完全不用管任何伺服器，按容器實際用到的資源計費。

📌 要不要管主機：完全不想管、想最省事 → Fargate；要自管 EC2 主機、要最多掌控 → EC2 啟動類型。

ECR：存放映像檔的私有倉庫 ​

ECR（Elastic Container Registry） 是 AWS 託管的私有映像檔倉庫。你把打包好的映像檔推（push）上去，ECS 或 EKS 要跑容器時就從這裡拉（pull）下來。它會保存與版本管理你的映像檔，和 IAM 權限整合，還能對映像檔做弱點掃描，檢查裡面的套件有沒有已知漏洞。

📌 ECR 是「放映像檔的地方」，ECS / EKS 是「用映像檔跑容器的地方」。

完整架構 ​

        ┌─ 編排器：ECS（AWS 原生） 或 EKS（Kubernetes）  ← 決定一
容器 ───┤
        ├─ 跑在哪：EC2 啟動類型（自管主機） 或 Fargate（免管主機）  ← 決定二
        │
        └─ 映像檔倉庫：ECR（ECS / EKS 從這裡拉映像）

把上面所有零件串成一條「從打包到上線」的路，這章就通了：

1. 打包    你的網站程式 → 做成 Docker 映像檔
2. 上傳    把映像檔 push 到 ECR（私有倉庫）
3. 定義    告訴 ECS：要跑哪個映像檔、給多少 CPU / 記憶體
4. 執行    ECS 在 Fargate 上啟動容器（主機 AWS 管，你不用碰）
5. 分流    前面掛一個 ALB，把使用者流量平均分到多個容器
6. 伸縮    流量大自動多開容器、流量小自動收掉

ECR 存映像檔、ECS 當調度中心、Fargate 出主機、ALB分流——一個容器化應用，就是這幾個服務各司其職。

容器 vs Lambda vs EC2：什麼時候選容器 ​

容器不是唯一的運算選擇，它夾在 EC2 與 Lambda 之間：

  • 要完全掌控作業系統、跑常駐或特殊工作 → EC2
  • 要短小、事件觸發、閒置不付錢 → Lambda
  • 要把應用打包成到處都能一致執行的單位、做微服務、或搬移既有的 Docker 應用上雲 → 容器（ECS / EKS）

重點整理 ​

  • 容器 vs 虛擬機：容器只打包「應用 + 相依套件」、共用主機 OS 核心，所以輕量、秒啟、好搬移；VM（如 EC2）各帶一份完整 OS，較重但隔離更強。兩者常搭配使用（容器跑在 VM 裡）。
  • 映像檔 vs 容器：映像檔是唯讀範本，容器是由映像檔啟動的執行實體；一個映像檔可啟動多個容器。
  • 編排（Orchestration） = 自動調度大量容器（排程、重啟、擴縮）。
  • 兩個要分開的決定：①用哪個編排器 ECS（AWS 原生、簡單、控制平面免費）vs EKS（開源 K8s、可跨雲、較複雜）；②容器跑在哪 EC2 啟動類型（自管主機、最多掌控）vs Fargate（免管主機、最省事）。
  • ECR = 私有映像檔倉庫，ECS / EKS 從這裡拉映像，支援版本管理與弱點掃描。
情境需求答案
要 AWS 原生、最簡單的容器編排ECS
要開源 Kubernetes、可跨雲 / 地端移植EKS
跑容器但完全不想管底層主機Fargate
跑容器且要自管 EC2 主機、要最多掌控EC2 啟動類型
存放與版本管理容器映像檔ECR
掃描映像檔裡的已知漏洞ECR（映像檔掃描）
把既有 Docker 應用一致地搬上雲容器（ECS / EKS）
短小、事件觸發的工作Lambda
