監控、合規與治理 ​

CloudWatch、CloudTrail、Config、Health 與 Trusted Advisor。

先備知識 ​

  • 監控與可觀測性（Monitoring / Observability） — 監控就是「持續量測系統健康、出狀況能馬上知道」。要先分清兩種資料：**指標（metrics）**是數字（如 CPU%、請求數），**日誌（logs）**是事件紀錄。這章的 CloudWatch 做的正是這件事，先有這個區分會好懂很多

    • 什麼是可觀測性？（Cloudflare，英文）
    • 監控與可觀測性（AWS，英文）

  • API 呼叫（API call） — 你在 AWS 做的每個操作（主控台點選、CLI、SDK）背後都是一次 API 呼叫。懂這點，才會理解 CloudTrail 為什麼能「記錄帳號裡發生的所有活動」——因為它記的就是這些 API 呼叫

    • 什麼是 API？（AWS，繁中）

  • 合規與稽核（Compliance / Audit） — 合規＝符合外部標準（如 SOC、PCI DSS、ISO 27001）；稽核＝留下證據證明你確實有遵守。這組概念撐起本章的 Config 合規檢查，以及 Artifact（下載報告）、Audit Manager（蒐集證據）

    • AWS 合規計畫（AWS，繁中）

為什麼要監控 ​

把服務丟上雲不是結束。你需要持續知道：資源跑得好不好？有沒有人動了不該動的東西？設定有沒有偏離規範？ 沒有監控，問題往往是「出事了才發現」，而且查不出是誰、什麼時候、改了什麼。

Amazon CloudWatch（效能與運作監控） ​

CloudWatch 回答的問題是：「我的資源現在表現如何？」它持續收集各種指標（metrics）（CPU 使用率、網路流量、請求數…），讓你設警報、看儀表板、集中日誌。

把它想成系統的健康監測儀——隨時顯示心跳、血壓（= CPU、流量等指標），數值異常就觸發警報叫人來。

功能做什麼
Metrics（指標）收集資源的效能數據（如 EC2 的 CPU 使用率）
Alarms（警報）指標超過門檻就動作，如「CPU > 80% → 發通知 / 觸發 Auto Scaling」
Dashboards（儀表板）把指標視覺化成圖表，一眼看懂系統狀態
Logs（日誌）集中收集應用程式與服務的日誌，方便查詢

📌 解題方向：「監控效能 / 設定門檻警報 / 視覺化指標」→ CloudWatch。CloudWatch Alarm 也常和 Auto Scaling、SNS 通知搭配。

AWS CloudTrail（活動稽核：誰做了什麼） ​

CloudTrail 回答的問題是：「是誰、在什麼時候、從哪裡、做了什麼動作？」它記錄帳號裡的每一個 API 呼叫——因為 AWS 上所有操作（主控台點選、CLI、SDK）本質都是 API 呼叫，所以等於記錄了所有活動。

把它想成大樓的監視器錄影 + 門禁日誌：誰幾點刷卡進來、開了哪扇門、做了什麼，全都留底。

  • 記錄內容：誰（哪個身分）、做了什麼動作（哪個 API）、什麼時間、來源 IP
  • 典型用途：資安鑑識（「是誰刪了那個 S3 Bucket？」）、稽核、合規追溯

📌 解題方向：「追查是誰做了某個操作 / 稽核帳號活動 / API 呼叫紀錄」→ CloudTrail。

AWS Config（組態記錄與合規檢查） ​

Config 回答的問題是：「我的資源設定長什麼樣？什麼時候被改過？符不符合規範？」它持續記錄資源的組態（configuration）與每次變更，還能用 Config Rules 自動檢查是否合規。

把它想成每個房間的設定快照與變更記錄簿：現在的擺設是什麼、上週長什麼樣、什麼時候被誰改成現在這樣、有沒有違反規定。

  • 典型用途：組態變更追蹤（「這個安全群組上週的規則是什麼？」）、合規檢查（「我所有的 EBS 磁碟區都加密了嗎？」「有沒有 S3 Bucket 被設成公開？」）

💡 還記得 M9 的伏筆嗎？「追蹤設定是否合規」就是 Config——不是只能下載報告的 Artifact。

🚨 核心辨別：CloudWatch vs CloudTrail vs Config ​

這三個最常一起出現逼你選，因為它們「都在看你的帳號」。用「回答什麼問題」來分最清楚：

CloudWatchCloudTrailConfig
回答的問題資源表現如何？誰做了什麼動作？設定是什麼 / 變了沒 / 合規嗎？
監看對象效能指標、日誌API 呼叫 / 帳號活動資源組態與變更
類比健康監測儀監視器 / 門禁日誌設定快照與變更記錄簿
典型用途警報、儀表板、效能監控稽核、追查誰改了東西合規檢查、組態變更追蹤

🚨 一句話記：效能看 CloudWatch、動作看 CloudTrail、組態與合規看 Config。

其他合規資源 ​

  • AWS Artifact：下載 AWS 的合規報告（SOC、PCI DSS、ISO…）—— M9 已介紹
  • AWS Audit Manager：自動蒐集證據、協助跑稽核流程

AWS Organizations（多帳號治理） ​

當你有很多 AWS 帳號（不同團隊、不同專案），需要集中治理。AWS Organizations 讓你把多個帳號統一管理：

功能做什麼
集中管理 / OU把帳號分組成組織單位（OU），統一套政策
SCP（服務控制政策）設帳號的權限上限（如「這些帳號只能用特定 Region」）
合併帳單（Consolidated Billing）多帳號一張帳單，並可合併用量享量大折扣

🚨 合併帳單是常考點：把多個帳號的用量加總，更容易達到量大折扣的門檻，整體更省。這也會在 M11 定價章再遇到。

📌 SCP 設的是上限 / 邊界，不是授權（最終權限 = IAM ∩ SCP）。

AWS Health（健康狀態） ​

AWS Health Dashboard 讓你知道「AWS 服務本身、以及你的資源」的健康狀況，分兩個視角：

視角看什麼
服務健康（Service health）整體 AWS 服務的狀態（哪個 Region 的哪個服務出狀況）
帳號健康（Your account health）影響到你自己資源的事件（如你的某台 EC2 即將進行排程維護）

🚨 辨別：「AWS 哪個服務掛了」→ 服務健康；「有什麼事會影響到我的資源」→ 帳號健康（原 Personal Health Dashboard）。

AWS Trusted Advisor（最佳實踐健檢） ​

Trusted Advisor 像一位自動的最佳實踐稽核員：掃描你的帳號，對照 AWS 最佳實踐，給出改善建議（紅 = 建議行動、黃 = 建議調查、綠 = 沒問題）。

它的檢查分成六大類：

類別檢查什麼
成本最佳化找出浪費的閒置資源（沒用的 EIP、低使用率 EC2）
效能找出可提升效能的設定
安全對照安全最佳實踐（如 root 帳號有沒有開 MFA、S3 權限）
容錯（Fault Tolerance）找單點故障（如 RDS 沒開 Multi-AZ、ASG 只在單一 AZ）
服務限額（Service Limits）用量接近帳號配額時提醒你
卓越營運（Operational Excellence）營運管理 hygiene（如是否用 Config 規則、Lambda runtime 是否過時）

💡 時效注意：許多舊教材還寫「5 大類」——那是舊的。「卓越營運」是 2023 年新增的第 6 類，考試請用 6 類。

🚨 三個「檢查」服務辨別：

  • Trusted Advisor = 對照 AWS 最佳實踐（涵蓋成本 / 效能 / 安全 / 容錯 / 限額 / 營運）
  • Inspector = 找軟體漏洞（M9）
  • Config = 檢查資源組態是否合規

考試重點整理 ​

  • 🚨 三大辨別（最重要）：CloudWatch（效能指標 / 警報 / 儀表板）、CloudTrail（誰做了什麼——API 活動稽核）、Config（設定是什麼 / 變了沒 / 合規嗎）。一句話：效能看 CloudWatch、動作看 CloudTrail、組態看 Config。
  • CloudWatch Alarm 常搭 Auto Scaling / SNS；CloudTrail 用於資安鑑識與稽核；Config 搭配 Config Rules 自動合規檢查
  • AWS Organizations：集中治理多帳號、SCP（權限上限）、合併帳單（量大折扣）
  • 🚨 AWS Health：服務健康（整體 AWS 狀態）vs 帳號健康（影響你資源的事件）
  • 🚨 Trusted Advisor：最佳實踐健檢，6 大類（成本、效能、安全、容錯、服務限額、卓越營運——第 6 類為 2023 新增，別再背成 5 類）
  • 🚨 三個「檢查」服務：Trusted Advisor（最佳實踐）、Inspector（軟體漏洞）、Config（組態合規）
情境需求答案
監控 EC2 的 CPU、設門檻警報CloudWatch
把效能指標做成儀表板CloudWatch Dashboards
追查「是誰刪了那個 Bucket」CloudTrail
稽核帳號的所有 API 活動CloudTrail
檢查「所有 EBS 是否都加密」、追蹤組態變更AWS Config
下載 PCI DSS / SOC 合規報告AWS Artifact
多帳號集中治理、設權限上限AWS Organizations（SCP）
多帳號合併帳單、享量大折扣Consolidated Billing
查 AWS 哪個服務目前有狀況AWS Health（服務健康）
查有什麼事件會影響我的資源AWS Health（帳號健康）
