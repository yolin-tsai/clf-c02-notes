## 網路在雲端裡是什麼

「網路」就是讓裝置之間能互相傳資料的連線。在 AWS 上，你不是把伺服器隨便丟到網際網路上，而是先圈出一塊屬於你、與其他人隔離的私人網路空間，再決定誰能進、誰能出、走哪條路。這塊空間就是 **VPC**。

> 把 VPC 想成：在 AWS 這座超大型園區裡，租下一塊用圍牆圈起來、只屬於你的私人園區——裡面的格局、道路、門禁，全部你自己規劃。把網路設計好，是雲端安全的第一道防線。

## VPC 與 Subnet：組織你的雲端資源

### VPC（Virtual Private Cloud，虛擬私有雲）

建立 VPC 時，你會指定一段私有 IP 範圍（用 CIDR 表示，例如 `10.0.0.0/16`）。VPC 與其他客戶、甚至你自己其他的 VPC，預設都是隔離的——別人進不來，你的資源也不會暴露在外。

### Subnet：把 VPC 再分區

把 VPC 切成更小的 Subnet，原因是不同資源需要不同的對外暴露程度。最經典的設計是「Public / Private」兩層：

```
VPC（10.0.0.0/16）── 你的私人園區
│
├─ Public Subnet   10.0.1.0/24
│    └─ Web Server（需要被網際網路訪問）
│
└─ Private Subnet  10.0.2.0/24
     └─ Database（絕不該讓網際網路直接碰到）
```

| Subnet 類型 | 說明 |
|---------|------|
| **Public Subnet** | 裡面的資源可以和網際網路來往（例如對外的網站伺服器）|
| **Private Subnet** | 裡面的資源不能被網際網路直接存取（例如資料庫），更安全 |

> **為什麼這樣設計？** 把「必須對外」的東西（網站）和「絕不該對外」的東西（資料庫）分開放，萬一網站被攻破，攻擊者也不會直接碰到資料庫。這是「縱深防禦」的基本盤。

> 📌 一個 Subnet 只屬於一個 AZ。要做上一章講的「跨多 AZ 高可用」，要在不同 AZ 各建一個 Subnet，把資源分散進去。

### Route Table（路由表）

每個 Subnet 都關聯一張路由表，像園區裡的「路標」，決定「要去某個目的地，該往哪條路走」。Public Subnet 的路由表會有一條規則：「要去網際網路 → 走 Internet Gateway」。Private Subnet 則沒有這條，流量出不去——這正是它「私有」的原因。

### NAT Gateway：讓 Private Subnet「只出不進」

Private Subnet 沒有往 Internet Gateway 的路由，所以流量出不去。但這帶來一個實際問題：Private Subnet 裡的資料庫或應用伺服器，有時還是需要**主動連出去**——例如下載安全性更新。難道為了更新，就要把它搬到 Public Subnet、暴露在網際網路上嗎？

不用。**NAT Gateway** 就是解法：它放在 Public Subnet，替 Private Subnet 的資源代為連出網際網路。

```
Private Subnet 的 DB ──（要更新，主動連出）──► NAT Gateway（在 Public Subnet）──► IGW ──► 網際網路
網際網路 ──（主動想連進來找 DB）──► ✗ 被擋下，進不來
```

關鍵在**方向**：
- ✅ Private Subnet「主動連出去」（下載更新）→ 通
- ❌ 網際網路「主動連進來」找 Private Subnet → 擋

> 💡 **IGW vs NAT Gateway**：Internet Gateway 是「雙向大門」（可進可出）；NAT Gateway 是「單向閘門」（Private Subnet 只能出、不能被進）。

## 連到 AWS 的方式

VPC 圈好後，決定「外界怎麼連進來、地端機房怎麼和 VPC 通」：

```
網際網路
    ↓
Internet Gateway（VPC 的對外大門）
    ↓
VPC（你的私人網路）
    ↑
VPN 或 Direct Connect（地端機房連進來的路）
```

| 連線方式 | 走公網嗎 | 特性 | 適合 |
|---------|---------|------|------|
| **Internet Gateway** | 是（公開）| 讓 VPC 連上公共網際網路 | 對外的網站、API |
| **Site-to-Site VPN** | 是（但加密）| 加密隧道、便宜、快上線、延遲不保證 | 臨時 / 成本敏感的地端連線 |
| **Direct Connect** | 否（專線）| 私有實體線路、穩定、低延遲、較貴；**不加密**† | 大量 / 敏感資料、長期穩定連線 |

> † **Direct Connect 不內建加密**：走專線代表流量不經公網，但線路本身沒有加密。需要加密時，得在 Direct Connect 上再疊一層 VPN（DX + VPN）。「私有專線 ≠ 自動加密」。

> **VPN vs Direct Connect（高頻辨別題）**：強調「**加密、便宜、快速建立**」→ VPN；強調「**穩定頻寬、低延遲、不走公網、專屬連線**」→ Direct Connect。兩者都能連地端，差別在「走公網加密」還是「實體專線」。

## VPC 之間的連通

### VPC Peering

兩個 VPC 直接互通，就像在兩個私人園區之間開一扇門。同一個 Region 或跨 Region 都支援。

關鍵限制：**VPC Peering 不可傳遞（non-transitive）**。

```
A ↔ B、B ↔ C，但 A ✗ C（A 不能透過 B 到達 C）
想讓 A 和 C 互通，必須再另建一條 A ↔ C 的 Peering
```

> 📌 「兩個 VPC 要互通」→ **VPC Peering**；「VPC 數量多、想集中管理互通」→ **Transit Gateway**。

### Transit Gateway

當 VPC 多起來，兩兩 Peering 的組合爆炸（10 個 VPC 要 45 條）。**Transit Gateway** 是一個中央交換機（hub），所有 VPC 和地端網路（VPN / Direct Connect）都接到它，形成 hub-and-spoke 架構，流量集中管理。

```
VPC A ─┐
VPC B ─┼─→ Transit Gateway ←─ 地端機房（VPN / Direct Connect）
VPC C ─┘
```

> 📌 「多個 VPC 集中互通 / 跨帳號 / 同時連地端」→ **Transit Gateway**。

### VPC Endpoints

讓 **Private Subnet 的資源直接存取 AWS 服務（如 S3）**，流量不離開 AWS 內部網路——不需要 Internet Gateway、也不需要 NAT Gateway。

CLF 最常考的是 **Gateway Endpoint**（免費）：

| 類型 | 支援服務 | 費用 |
|------|---------|------|
| **Gateway Endpoint** | S3、DynamoDB | 免費 |
| Interface Endpoint（PrivateLink）| 其他大多數 AWS 服務 | 有費用 |

> 📌 「Private Subnet 要存取 S3，但不想走公網 / 不想過 NAT Gateway」→ **VPC Endpoint（Gateway）**。

## 雲端安全：Security Group 與 NACL

資源放進 Subnet 後，要控制「什麼流量能進、能出」。AWS 提供兩層防火牆，是這章——乃至整個安全領域——最常考、最容易混淆的地方。

```
網際網路
    ↓
Internet Gateway（VPC 對外大門）
    ↓
Route Table（路標：決定流量往哪走）
    ↓
NACL（Subnet 邊界的「社區哨點」：守整個 Subnet）
    ↓
Security Group（執行個體門口的「保全」：守單一台機器）
    ↓
EC2 執行個體
```

### Security Group（安全群組）：執行個體層級、有狀態

- **執行個體層級**：直接掛在 EC2 上，是最貼近機器的那道牆
- **有狀態（Stateful）**：允許某個流量進來，它的**回應流量自動放行**，不用另開規則（保全記得「這個人是我放進來的，要出去當然放行」）
- **只能寫允許規則**：沒被允許的自動擋掉，無法寫「拒絕」

### Network ACL（NACL）：Subnet 層級、無狀態

- **Subnet 層級**：守在 Subnet 邊界，Subnet 內所有資源都受它管
- **無狀態（Stateless）**：進、出**分開獨立檢查**，你允許了「進來」，不代表回應能「出去」——必須另外明確允許回應流量（哨點不記得誰剛進來，出去的車要另外查一遍）
- **可以寫允許也可以寫拒絕**：依規則編號由小到大逐條比對，適合用來明確封鎖某些 IP

### Security Group vs NACL

| 比較 | Security Group | Network ACL（NACL）|
|------|---------------|-------------------|
| **守備層級** | 執行個體（單台機器）| Subnet（整個 Subnet 邊界）|
| **狀態** | **有狀態**：回應自動放行 | **無狀態**：進出分開、回應要另開規則 |
| **規則類型** | 只有**允許**（allow）| **允許 + 拒絕**（allow / deny）|
| **規則比對** | 全部規則一起看 | 依編號由小到大逐條，先符合先生效 |
| **預設行為** | 入站全擋、出站全通 | 自建的全擋（要自己加規則）|
| **像什麼** | 機器門口的保全 | 社區外圍的哨點 |

> 1. **層級**：Security Group 在執行個體、NACL 在 Subnet
> 2. **狀態**：Security Group 有狀態（回應自動通）、NACL 無狀態（回應要自己開）
> 3. **拒絕規則**：只有 NACL 能明確「拒絕」；Security Group 只能「允許」
>
> 📌 看到「要**封鎖某個特定 IP 位址**」→ 答案是 **NACL**（Security Group 寫不出拒絕規則）。

## 全球網路：Route 53 與 CloudFront

最後兩個服務，把流量「在全球範圍內」導到對的地方、送得更快。

### Amazon Route 53（DNS 服務）

Route 53 是 AWS 的 **DNS（Domain Name System，網域名稱系統）** 服務。DNS 是「網路的電話簿」——使用者輸入 `www.example.com`，DNS 把這個名字翻成伺服器的 IP 位址，瀏覽器才知道要去哪裡。

Route 53 不只做翻譯，還能：
- **註冊網域**：直接在 AWS 買網域名稱
- **路由政策**：依延遲導到最快的據點、依地理位置把不同國家使用者導到不同伺服器、加權分配流量

> 💡 Route 53 常和 CloudFront、ELB 搭配：DNS 先把使用者導到最近 / 最適合的入口，後面再由 CDN 或負載平衡器接手。

### Amazon CloudFront（CDN）

CloudFront 是 AWS 的 **CDN（Content Delivery Network，內容傳遞網路）**。

```
使用者（台北）→ 最近的 Edge Location
                ├─（快取有）→ 直接回應，超快
                └─（快取沒有）→ 回源頭（origin，如 S3 / EC2）拿，再快取
```

- 適合：圖片、影片、靜態網站、串流等需要全球快速傳遞的內容
- 好處：大幅降低延遲、減輕源頭伺服器負擔

> 📌 **解題方向**：「把內容快取到離全球使用者最近處、降低延遲」→ **CloudFront**；「把網域名稱解析成 IP / 依延遲或地理導流」→ **Route 53**。

## 封包的完整旅程

回到開頭的目標——一個封包怎麼從網際網路走到一台 EC2。現在你手上已經有全部零件，把它們接起來：

```
使用者請求
   ↓
Internet Gateway   VPC 的對外大門
   ↓
Route Table        路標：這段流量該送去哪個 Subnet
   ↓
NACL               Subnet 哨點：這個來源 / Port 允不允許進？（無狀態）
   ↓
Security Group     機器保全：這台 EC2 允不允許這個流量？（有狀態）
   ↓
EC2 執行個體       抵達目的地
```

1. 使用者請求從網際網路進來，先到 **Internet Gateway**。
2. **Route Table** 決定流量送往哪個 Subnet。
3. 到了 Subnet 邊界，**NACL** 逐條檢查允不允許進。（無狀態，稍後回應還要再查一次。）
4. 來到 EC2 門口，**Security Group** 確認允不允許這個流量。（有狀態，回應自動放行。）
5. 兩道牆都放行，封包才真正抵達 **EC2**。

**回程**：EC2 回應時，Security Group 有狀態自動放行；但 NACL 無狀態，必須另外開好「允許回應出去」的規則，回應才出得了 Subnet。

## 重點整理

- **VPC** = 你在 AWS 上隔離的私人網路，建立時指定 CIDR（如 `10.0.0.0/16`）
- Subnet 分**公有**（可連網際網路）與**私有**（不可被外網直接存取，放資料庫等）；一個 Subnet 只屬於**一個 AZ**
- **Route Table** 決定 Subnet 流量往哪走（Public Subnet 有「往 IGW」的路由，Private Subnet 沒有）
- **NAT Gateway** = 讓 Private Subnet「只出不進」地連網際網路（如下載更新）
- 連線三方式：**Internet Gateway**（連公網）、**VPN**（公網加密隧道、便宜快速）、**Direct Connect**（實體專線、穩定低延遲、不走公網、較貴）
- **Security Group vs NACL** 四組對比：
  1. 層級：Security Group 在執行個體、NACL 在 Subnet
  2. 狀態：Security Group 有狀態（回應自動放行）、NACL 無狀態（回應要另開規則）
  3. 規則：Security Group 只能允許；NACL 可允許可拒絕（依編號由小到大）
  4. 預設：Security Group 入站全擋、出站全通；自建 NACL 全擋、要自己加規則
- **Route 53** = DNS 服務：網域解析成 IP、註冊網域、依延遲 / 地理 / 加權路由
- **CloudFront** = CDN：用 Edge Location 快取內容、降低全球延遲
- **VPC Peering**：兩個 VPC 直接互通；不可傳遞（A↔B、B↔C，A 不通 C）
- **Transit Gateway**：多 VPC hub-and-spoke 集中互通，也能連地端
- **VPC Endpoint（Gateway）**：Private Subnet 不走公網直接存取 S3 / DynamoDB，免費

| 情境需求 | 答案 |
|---------|------|
| 在雲端圈一塊隔離的私人網路 | **VPC** |
| 資料庫不該被網際網路直接存取 | 放**Private Subnet** |
| 讓 VPC 連上公共網際網路 | **Internet Gateway** |
| Private Subnet 要下載更新，但不被外網存取 | **NAT Gateway** |
| 地端連 AWS、加密、便宜、快上線 | **VPN** |
| 地端連 AWS、專線、穩定低延遲 | **Direct Connect** |
| 控制單一台 EC2 的進出流量 | **Security Group**（有狀態）|
| 在 Subnet 邊界明確封鎖某個 IP | **NACL**（無狀態、可寫拒絕規則）|
| 把網域名稱解析成 IP / 智慧導流 | **Route 53** |
| 把內容快取到離使用者最近處降延遲 | **CloudFront** |
| 兩個 VPC 互通 | **VPC Peering** |
| 多個 VPC 集中互通 / 同時連地端 | **Transit Gateway** |
| Private Subnet 存取 S3 不走公網 | **VPC Endpoint（Gateway）** |
