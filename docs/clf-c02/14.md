# 監控、合規與治理

CloudWatch、CloudTrail、Config、Health 與 Trusted Advisor。

## 為什麼要監控

把服務丟上雲不是結束。你需要持續知道：資源跑得好不好？有沒有人動了不該動的東西？設定有沒有偏離規範？沒有監控，問題往往是「出事了才發現」，而且查不出是誰、什麼時候、改了什麼。

## Amazon CloudWatch（效能與運作監控）

CloudWatch 回答的問題是：「我的資源現在表現如何？」它持續收集各種**指標（metrics）**（CPU 使用率、網路流量、請求數…），讓你設**警報**、看**儀表板**、集中**日誌**。

把它想成系統的**健康監測儀**——隨時顯示心跳、血壓（= CPU、流量等指標），數值異常就觸發警報叫人來。

| 功能 | 做什麼 |
|------|--------|
| **Metrics（指標）** | 收集資源的效能數據（如 EC2 的 CPU 使用率）|
| **Alarms（警報）** | 指標超過門檻就動作，如「CPU > 80% → 發通知 / 觸發 Auto Scaling」|
| **Dashboards（儀表板）** | 把指標視覺化成圖表，一眼看懂系統狀態 |
| **Logs（日誌）** | 集中收集應用程式與服務的日誌，方便查詢 |

> 📌 **解題方向**：「監控效能 / 設定門檻警報 / 視覺化指標」→ **CloudWatch**。CloudWatch Alarm 也常和 Auto Scaling、SNS 通知搭配。

## AWS CloudTrail（活動稽核）

CloudTrail 回答的問題是：「是誰、在什麼時候、從哪裡、做了什麼動作？」它記錄帳號裡的**每一個 API 呼叫**——因為 AWS 上所有操作（主控台點選、CLI、SDK）本質都是 API 呼叫，所以等於記錄了所有活動。

把它想成大樓的**監視器錄影 + 門禁日誌**：誰幾點刷卡進來、開了哪扇門、做了什麼，全都留底。

- **記錄內容**：誰（哪個身分）、做了什麼動作（哪個 API）、什麼時間、來源 IP
- **典型用途**：資安鑑識（「是誰刪了那個 S3 Bucket？」）、稽核、合規追溯

> 📌 **解題方向**：「**追查是誰做了某個操作 / 稽核帳號活動 / API 呼叫紀錄**」→ **CloudTrail**。

## AWS Config（組態記錄與合規檢查）

Config 回答的問題是：「我的資源設定長什麼樣？什麼時候被改過？符不符合規範？」它持續記錄資源的組態（configuration）與每次變更，還能用 **Config Rules** 自動檢查是否合規。

把它想成每個房間的**設定快照與變更記錄簿**：現在的擺設是什麼、上週長什麼樣、什麼時候被誰改成現在這樣、有沒有違反規定。

- **用途**：組態變更追蹤（「這個安全群組上週的規則是什麼？」）、**合規檢查**（「我所有的 EBS 磁碟區都加密了嗎？」「有沒有 S3 Bucket 被設成公開？」）

| | CloudWatch | CloudTrail | Config |
|---|-----------|-----------|--------|
| 回答的問題 | 資源表現如何？ | 誰做了什麼動作？ | 設定是什麼 / 變了沒 / 合規嗎？ |
| 監看對象 | 效能指標、日誌 | API 呼叫 / 帳號活動 | 資源組態與變更 |
| 類比 | 健康監測儀 | 監視器 / 門禁日誌 | 設定快照與變更記錄簿 |
| 典型用途 | 警報、儀表板、效能監控 | 稽核、追查誰改了東西 | 合規檢查、組態變更追蹤 |

> **效能看 CloudWatch、動作看 CloudTrail、組態與合規看 Config。**

## 其他合規資源

- **AWS Artifact**：下載 AWS 的合規報告（SOC、PCI DSS、ISO…）
- **AWS Audit Manager**：自動蒐集證據、協助跑稽核流程

## AWS Organizations（多帳號治理）

當你有很多 AWS 帳號（不同團隊、不同專案），需要**集中治理**。AWS Organizations 讓你把多個帳號統一管理：

| 功能 | 做什麼 |
|------|--------|
| **集中管理 / OU** | 把帳號分組成組織單位（OU），統一套政策 |
| **SCP（服務控制政策）** | 設帳號的**權限上限**（如「這些帳號只能用特定 Region」）|
| **合併帳單（Consolidated Billing）** | 多帳號**一張帳單**，並可**合併用量享量大折扣** |

> 📌 **SCP** 設的是上限 / 邊界，不是授權（最終權限 = IAM ∩ SCP）。

## AWS Trusted Advisor（最佳實踐健檢）

Trusted Advisor 像一位**自動的最佳實踐稽核員**：掃描你的帳號，對照 AWS 最佳實踐，給出改善建議（紅 = 建議行動、黃 = 建議調查、綠 = 沒問題）。

它的檢查分成**六大類**：

| 類別 | 檢查什麼 |
|------|---------|
| **成本最佳化** | 找出浪費的閒置資源（沒用的 EIP、低使用率 EC2）|
| **效能** | 找出可提升效能的設定 |
| **安全** | 對照安全最佳實踐（如 root 帳號有沒有開 MFA、S3 權限）|
| **容錯（Fault Tolerance）** | 找單點故障（如 RDS 沒開 Multi-AZ、ASG 只在單一 AZ）|
| **服務限額（Service Limits）** | 用量接近帳號配額時提醒你 |
| **卓越營運（Operational Excellence）** | 營運管理 hygiene（如是否用 Config 規則、Lambda runtime 是否過時）|

**三個「檢查」服務辨別**：
- **Trusted Advisor** = 對照 **AWS 最佳實踐**（涵蓋成本 / 效能 / 安全 / 容錯 / 限額 / 營運）
- **Inspector** = 找軟體漏洞
- **Config** = 檢查資源組態是否合規

## 重點整理

- **CloudWatch**（效能指標 / 警報 / 儀表板）、**CloudTrail**（誰做了什麼——API 活動稽核）、**Config**（設定是什麼 / 變了沒 / 合規嗎）。
- **CloudWatch** Alarm 常搭 Auto Scaling / SNS；**CloudTrail** 用於資安鑑識與稽核；**Config** 搭配 Config Rules 自動合規檢查
- **AWS Organizations**：集中治理多帳號、**SCP**（權限上限）、**合併帳單**（量大折扣）
- **Trusted Advisor**：最佳實踐健檢

| 情境需求 | 答案 |
|---------|------|
| 監控 EC2 的 CPU、設門檻警報 | **CloudWatch** |
| 把效能指標做成儀表板 | **CloudWatch Dashboards** |
| 追查「是誰刪了那個 Bucket」 | **CloudTrail** |
| 稽核帳號的所有 API 活動 | **CloudTrail** |
| 檢查「所有 EBS 是否都加密」、追蹤組態變更 | **AWS Config** |
| 下載 PCI DSS / SOC 合規報告 | **AWS Artifact** |
| 多帳號集中治理、設權限上限 | **AWS Organizations（SCP）** |
| 多帳號合併帳單、享量大折扣 | **Consolidated Billing** （Organizations）|
| 對照最佳實踐做帳號健檢（含成本 / 安全）| **Trusted Advisor** |
