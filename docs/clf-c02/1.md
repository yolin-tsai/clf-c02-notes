# 雲端入門

Amazon Web Services（AWS）是全球最大的雲端平台，提供運算、儲存、資料庫、網路等數百種「隨開即用」的服務。
這章是雲端的定義、服務分層方式、實體基礎設施、安全責任的劃分，後面每個章節都會用到這些基礎。

## 什麼是雲端運算

傳統公司要架設一套系統，得先買伺服器、租機房、請人維護，所有費用在上線前就得付清——還要猜「未來需要多大容量」。猜多了，機器閒置燒錢；猜少了，流量一來就垮掉，再追加也來不及。

雲端把這個賭注消掉了。**雲端運算是「透過網際網路按需取用 IT 資源、依用量付費」**——就像用電是按度計費，你家不需要自己買發電機。

雲端最核心的財務優勢，就是把 IT 支出從 **CapEx** 轉變成 **OpEx**：

| 名詞 | 意思 | 例子 |
|------|------|------|
| **CapEx**（Capital Expenditure，資本支出）| 先砸一大筆錢購置設備，費用一次性發生 | 買伺服器、蓋機房 |
| **OpEx**（Operating Expenditure，營運支出）| 用多少付多少的浮動費用 | 按月付 AWS 帳單 |

以學校選課系統為例：開學前兩天是選課高峰，幾萬人同時搶課；其餘 90% 的時間幾乎沒有流量。如果學校自建機房（CapEx），得按尖峰容量一次買足夠多的伺服器，但這些機器平常閒著白白燒電費。換成雲端（OpEx），選課期間自動擴大、結束就縮回，費用完全跟著實際用量走。

### 三種部署模型

要多少自己建、要多少委外，取決於公司的情況。這就有了三種選擇：

| 模型 | 說明 | 適合 |
|------|------|------|
| **雲端（Cloud）** | 全部跑在 AWS 等雲端平台 | 新創、雲端原生應用 |
| **地端（On-premises）** | 全部在自己機房，又叫私有雲 | 法規嚴格、已有大量本地投資 |
| **混合（Hybrid）** | 部分在雲、部分在本地，兩邊連通 | 逐步上雲、特定資料須留在本地 |

### 三種服務模型

使用雲端時，「你自己要管多少」可以有不同程度的選擇。愈往 SaaS 走，AWS 幫你管的東西愈多，你能控制的細節也愈少：

| 模型 | 你管什麼 | AWS 管什麼 |
|------|---------|-----------|
| **IaaS**（基礎設施即服務）| OS 以上全部（自己裝系統、設定環境）| 硬體、網路、虛擬化 |
| **PaaS**（平台即服務）| 只管程式碼與資料 | OS、執行環境 |
| **SaaS**（軟體即服務）| 直接用，幾乎什麼都不管 | 幾乎全部 |

用吃披薩來記這個光譜：IaaS 是租店家的烤箱自帶食材，PaaS 是買冷凍披薩回家烤（飲料和餐桌自己準備），SaaS 是直接上餐廳——管的事愈來愈少，自由度也愈來愈低。

延伸閱讀：[IaaS vs. PaaS vs. SaaS（Red Hat）](https://www.redhat.com/en/topics/cloud-computing/iaas-vs-paas-vs-saas) 


## 雲端的六大好處

AWS 定義了採用雲端的六大優勢，考試常給情境問「這體現了哪個好處」。先看下表掌握對應關鍵字，再讀各項說明：

| # | 英文 | 好處 | 情境 |
|---|---------|------|-----------|
| 1 | Trade Fixed Expense for Variable Expense | 不先砸錢買設備，用多少付多少 | 「不想先投入大筆資本」|
| 2 | Benefit from Massive Economies of Scale | 跟 AWS 百萬客戶共享低採購成本 | 「降低單位成本」|
| 3 | Stop Guessing Capacity | 要多少開多少，不用猜 | 「不確定流量」「避免猜錯容量」|
| 4 | Increase Speed and Agility | 幾分鐘部署，不用等採購 | 「快速實驗 / 上線」|
| 5 | Stop Spending on Data Centers | 不必管機房、電力、硬體 | 「專注本業，不想管基礎設施」|
| 6 | Go Global in Minutes | 幾個步驟就能在全球各地上線 | 「服務全球使用者 / 降低延遲」|

### 1. Trade Fixed Expense for Variable Expense（以變動支出取代固定支出）

「固定支出」的意思是：不管這筆投資將來划不划算，你都得在**第一天先付清一大筆錢**。對一個還沒有任何使用者、還不知道會不會成功的新產品來說，這是一場豪賭——押對了，當初買的機器剛好夠用；押錯了，要嘛買太多、機器整年閒著折舊，要嘛買太少、真的紅了卻擴充不及。

雲端把這場豪賭換成「變動支出」：費用跟著**實際用量**走。一個乏人問津的專案幾乎不花錢，一個成長中的專案則隨規模按比例付費——你永遠不會在拿到價值之前先付錢。這也大幅拉低了「試新東西」的門檻：想驗證一個點子，成本從一張伺服器採購單，變成幾個小時的運算費用。

### 2. Benefit from Massive Economies of Scale（享受規模經濟）

規模經濟的運作原理：當數十萬個客戶的用量被集中到同一個平台，AWS 採購硬體、電力與網路的規模，是任何單一公司都達不到的，於是每單位成本被壓到極低；又因為這是一個按用量計價、彼此競爭的市場，這些省下來的成本會透過**更低的價格**回饋給每個客戶。

可以想成量販店：一次替幾百萬人採買，拿到的單價是一個家庭自己去買永遠談不到的。

### 3. Stop Guessing Capacity（免除預估容量）

自建機房困難的地方在於：必須在**資訊最少的時候**（上線前）就決定要買多少容量。猜的結果只有兩種壞法——猜多了，一整排伺服器閒著燒電費與折舊；猜少了，流量一來就垮，而追加實體伺服器要等好幾週，根本來不及救。

雲端直接把「猜」這個動作拿掉：現在需要多少就開多少，再搭配 **Auto Scaling** 讓資源隨實際需求自動增減。回到前面的選課系統——選課那兩天自動放大、結束就縮回，全程不需要事先預測尖峰有多高。這種「隨需求彈性伸縮」的能力，是雲端最具代表性的優勢。

### 4. Increase Speed and Agility（提升速度與敏捷性）

過去要拿到一台伺服器，得走採購流程、下訂、等貨、上架安裝，前後往往數週到數月。這條長長的前置時間有個隱藏代價：因為「開一台機器」又慢又貴，團隊會傾向**不敢輕易嘗試**有風險的新點子。

在 AWS 上佈建環境只要幾分鐘，而且開得起、也拆得起。當「試一次」的成本趨近於零，團隊的行為就變了：可以放手實驗、便宜地失敗、快速迭代。敏捷性的重點不只是「快」，更是這種低實驗成本帶來的**放膽去試的自由**。

### 5. Stop Spending Money on Running and Maintaining Data Centers（省下維運機房的費用）

自己經營一座資料中心，是一筆和你真正的產品**毫無關係**的持續開銷：供電、散熱、實體門禁、定期更換故障硬體，還要專人負責上架伺服器、半夜換掉壞掉的硬碟。這類工作 AWS 稱為「undifferentiated heavy lifting（無差異化的粗重工作）」——做得再好也不會讓你的產品更有競爭力。

把這些全部交給 AWS，你的工程人力就能專注在真正能做出差異的地方——你的應用本身。對一間新創來說，差別就是工程師在寫功能，而不是去機房換硬碟。

### 6. Go Global in Minutes（數分鐘內全球佈署）

用傳統做法要服務另一個大洲的使用者，你得在當地找地、租機房、把設備一路建起來——好幾個月的工程與龐大成本，以致多數公司根本做不到。

有了 AWS 遍布全球的 Region，把服務擴展到新的地理區只需要幾個步驟，使用者就能連到離自己最近的基礎設施，延遲更低、體驗更穩。這等於把「拓展到全球」從一項大型資本工程，變成了一個**設定選項**。舉例來說，一個台灣的應用想服務美國使用者，只要多開一個美國 Region，美國那端就不必再繞過整個太平洋來連線。

## AWS 全球基礎設施

AWS 把實體設施分成三個層級，設計這些層級的原因不只是地理方便，更是為了隔離故障：

```
Region（區域，如 ap-northeast-1 東京）
  └─ Availability Zone（可用區，一個 Region 通常 ≥ 3 個）
       └─ 一或多個實體資料中心
（另外散布全球的）Edge Location（邊緣節點）→ 給 CloudFront 做快取
```

| 層級 | 是什麼 | 重點 |
|------|--------|------|
| **Region（區域）** | 全球各地的地理區域 | 彼此**獨立隔離**；資料預設不自動跨 Region 移動 |
| **Availability Zone（可用區）** | Region 內獨立的資料中心群 | 實體隔離（不同電力 / 機房），但以低延遲高速網路相連 |
| **Edge Location（邊緣節點）** | 散布全球的快取點，數量遠多於 Region | CloudFront 用它把內容放到離使用者最近的地方 |

為什麼一個 Region 要拆成多個實體隔離的 Availability Zone？因為這樣一個機房失火或斷電，另一個 Availability Zone 還活著——把應用程式跨多個 Availability Zone 部署，就是高可用架構的基礎。

> 📌 **全域服務 vs 區域服務**：EC2、S3、RDS 等絕大多數服務都是 **Region 範圍**，建立資源時要選 Region。但有少數服務是**全域**的，不需選 Region——**IAM**（帳號權限）、**Route 53**（DNS）、**CloudFront**（CDN）、**帳單（Billing）** 設定在所有 Region 共用。

### 如何選擇 Region

選 Region 要綜合四個因素考量：

1. **法規與資料落地（Compliance）** — 法規要求資料留在特定國家 → 選該國 Region，此項優先於其他考量
2. **貼近使用者（Proximity）** — 使用者在哪，就選近的 Region 降低延遲
3. **服務可用性（Feature availability）** — 新服務不是每個 Region 都有，要先確認
4. **價格（Pricing）** — 不同 Region 定價不同

## 責任共擔模型（Shared Responsibility Model）

使用 AWS 不代表安全都交給 AWS 處理，AWS 和你是分工關係：

```
AWS  負責「雲本身的安全」（Security OF the Cloud）
你   負責「雲裡面的安全」（Security IN the Cloud）
```

| 誰負責 | 範圍 |
|--------|------|
| AWS | 實體資料中心、硬體、底層網路、虛擬化層 |
| 你 | 資料、IAM 權限設定、OS 與防火牆（EC2）、加密設定、應用程式 |

責任界線隨服務類型移動——服務愈託管，AWS 扛的愈多。以「系統更新」為例：

- EC2（IaaS）：你有 root 權限，OS 是你的地盤，更新得自己來
- RDS（託管服務）：資料庫引擎的更新由 AWS 負責，你只管資料與存取權限

> 📌 無論用哪種服務，**資料**與 **IAM 權限設定**永遠是客戶的責任。

## 重點整理

- **雲端運算** = 透過網路按需取用 IT 資源、依用量付費
- 部署模型：雲端 / 地端 / 混合
- 服務模型：**IaaS**（自管 OS，如 EC2）、**PaaS**（只管程式碼，如 Beanstalk）、**SaaS**（直接用，如 Gmail）；愈往 SaaS 走 AWS 扛愈多
- 六大好處能對應情境關鍵字；最重要：**不必再預估容量**（對應 Auto Scaling）
- 全球基礎設施三層：**Region**（獨立地理區域）> **AZ**（實體隔離但低延遲相連的機房群）> **Edge Location**（CloudFront 快取點，數量最多）
- 跨多個 **AZ** 部署 = 高可用的基礎（一個 AZ 掛了還有別的）
- 選 Region 四考量：**法規落地 > 貼近使用者 > 服務可用性 > 價格**
- **全域服務**：**IAM、Route 53、CloudFront、帳單**；其他絕大多數服務都是 Region 範圍
- **責任共擔**：AWS 管「雲本身（OF）」，客戶管「雲裡面（IN）」；OS 補丁：EC2 是客戶責任、RDS 是 AWS 責任
- 資料與 IAM 權限設定永遠是客戶責任，與服務類型無關
